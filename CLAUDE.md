- the packet-tunnel provider is one set of shared sources (`Sources/PacketTunnel` + the `PacketTunnelBase` target template) shipped two ways: an app extension on iOS (`PacketTunnel` target, uses `NSExtensionMain`) and a system extension on macOS (`PacketTunnelSysEx` target, needs its own entry point in `Sources/PacketTunnelSysEx/main.swift` calling `NEProvider.startSystemExtensionMode()`). System-extension packaging is mandatory for Developer ID distribution outside the App Store. The app embeds the right one per platform via `destinationFilters`; the macOS sysex bundle id stays `$(BUNDLE_ID_PREFIX).PacketTunnel` (matches `TunnelsManager.providerBundleID` and satisfies the "sysex id is app-id-prefixed" rule). macOS app + sysex are non-sandboxed with Hardened Runtime; the app carries `com.apple.developer.system-extension.install` and activates the sysex via `SystemExtensionManager` (`OSSystemExtensionRequest`) â€” activation only works from `/Applications`, and sysextd does NOT replace an installed extension with an identical `CFBundleShortVersionString`/`CFBundleVersion`, so bump `CURRENT_PROJECT_VERSION` in project.yml when iterating on extension code or the old binary keeps running. Also: the sysex bundle must be named after its bundle id (`PRODUCT_NAME = $(PRODUCT_BUNDLE_IDENTIFIER)`), needs `NSSystemExtensionUsageDescription` in its Info.plist, and only loads from a notarized app. The sysex is a root daemon: it cannot use the data-protection keychain (no user context), so the app passes the auth token via `startTunnel(options:)` and the provider persists it in the System keychain (see AuthTokenKeychain); never put secrets in providerConfiguration (`/Library/Preferences/com.apple.networkextension.plist` is world-readable)
