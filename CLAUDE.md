- the Rust artifact (libezvpn.xcframework) is delivered via a local Swift package `Packages/Ezvpn/Package.swift` (referenced by local path in project.yml); its binary target downloads the pinned release zip by URL+checksum (reproducible; SPM auto-exposes the embedded ezvpn.h to the bridging header, so no vendor/ dir and no HEADER_SEARCH_PATHS). Bump to a new release with `scripts/bump-xcframework.sh <tag>` (rewrites url+checksum and sets both app/extension `MARKETING_VERSION` values from the tag) after the ezvpn release workflow publishes the `libezvpn-apple.xcframework.zip` asset.
- for FFI dev against a local Rust build, run `./build-apple.sh release` in the sibling then set `EZVPN_LOCAL_XCFRAMEWORK=1` for `xcodegen generate` and every `xcodebuild`/Xcode invocation — only the exact value `1` opts in. `Packages/Ezvpn/Package.swift` then links the local build via the committed symlink `Packages/Ezvpn/local/libezvpn.xcframework`, which points at sibling `../ezvpn/dist/apple/libezvpn.xcframework`, instead of the pinned release. Unset it (the default), or use any other value, to go back to the reproducible release zip. `scripts/run-device-ios.sh` and `scripts/run-macos.sh` set the var for you (local by default; `--pinned` to opt out)
- a Packet Tunnel Provider can't run in the iOS Simulator, so the xcframework carries `ios-arm64` and native `macos-arm64` slices only; verify with generic iOS and `platform=macOS,arch=arm64` builds
- the packet-tunnel provider is one set of shared sources (`Sources/PacketTunnel` + the `PacketTunnelBase` target template) shipped two ways: an app extension on iOS (`PacketTunnel` target, uses `NSExtensionMain`) and a system extension on macOS (`PacketTunnelSysEx` target, needs its own entry point in `Sources/PacketTunnelSysEx/main.swift` calling `NEProvider.startSystemExtensionMode()`). System-extension packaging is mandatory for Developer ID distribution outside the App Store. The app embeds the right one per platform via `destinationFilters`; the macOS sysex bundle id stays `$(BUNDLE_ID_PREFIX).PacketTunnel` (matches `TunnelsManager.providerBundleID` and satisfies the "sysex id is app-id-prefixed" rule). macOS app + sysex are non-sandboxed with Hardened Runtime; the app carries `com.apple.developer.system-extension.install` and activates the sysex via `SystemExtensionManager` (`OSSystemExtensionRequest`) — activation only works from `/Applications`
- macOS distribution: `scripts/create-archive-macos.sh` archives → exports Developer-ID-signed → notarizes (App Store Connect API key via notarytool) → staples → builds a drag-to-Applications `.dmg`. Needs a "Developer ID Application" cert and notary credentials set up manually on the Apple side first. `-m debugging` gives a local dev-signed `.app` with no notarization/dmg
